Assessor guidance

How this Trust Center works.

A short orientation for assessors. It explains what you are looking at, how each verdict is produced, and how to use it to run an efficient, evidence-based assessment, without taking anyone's word for the result.

◈What a Key Security Indicator is

FedRAMP 20x replaces long narrative control write-ups with Key Security Indicators (KSIs): concrete, testable statements of security posture grouped into ten families (Identity, Cloud-Native Architecture, Monitoring, Change Management, and so on). Instead of asserting "we encrypt data", a KSI is proven by machine-readable evidence pulled straight from the running system. This Trust Center is that proof surface for the offering, refreshed every run.

↻Where the numbers come from

Nothing on this page is entered by a person. On every scheduled run a collector executes a read-only query, the result is checked by a shared assertion, and the outcome is written into a signed Security Decision Record (SDR) that this page renders. The Assessor tab shows you that whole chain for each indicator, so a verdict is never something you have to trust blind, you can read the query and the logic that produced it.

⛓The validation pipeline

The same five steps run for every automated indicator. This is exactly what the Replay cycle button animates in Assessor mode.

1
Collector queries the environmentA read-only query runs against Azure Resource Graph, Microsoft Graph, Sentinel, or GitHub. Read-only means the act of assessing can never change the thing being assessed.
2
Evidence is capturedThe rows the query returns are saved verbatim with a timestamp and the subscription they came from. That payload is the evidence, and you can see it in the Input stage.
3
Assertion evaluates the evidenceA single Python function in engine/checks.py decides pass or fail. The same function feeds both the SDR and this page, so the two can never disagree.
4
Verdict is recordedPass, fail, or pending is written to the SDR. Pending means not measured (a permission or table was missing) and is never silently turned into a fail.
5
Trust Center republishesThis page is regenerated from the latest SDR, so the posture you read is the posture the last run actually proved.

◐The three assurance modes

Not every indicator can be settled by a machine. Each one carries a mode that tells you how much human judgement is still required, and what your job as an assessor is.

Auto

Fully machine-decided from live evidence. These are the only indicators that count toward the headline Verified number. Your job: sanity-check the query and the assertion.

Hybrid

An automated leg gathers evidence, but a person must still review and sign off. Shown as Review. Your job: confirm the attestation exists and the evidence supports it.

Manual

No automated test is possible (policy, training, executive review). Settled entirely by a dated attestation. Your job: verify the attestation and its review cadence.

✓Reading a verdict

Every indicator resolves to one of five states. The color always means the same thing, in any theme.

VALIDATEDAuto indicator whose assertion passed against live evidence.
FAILEDMeasured, and the evidence did not meet the pass criteria. A real finding.
REVIEWHybrid indicator awaiting or carrying human sign-off.
PENDINGNot yet measured (collector staged or permission missing). Not a fail.
MANUALSettled by attestation only, no automated test applies.

◆Verified vs attested

These are counted separately on purpose, and it matters when you read the headline.

  • Verified counts only Auto indicators whose assertion currently passes. That is the number of things a machine can prove right now, with no human in the loop.
  • Attested counts indicators where a named person has signed off a review. It never inflates the verified count, an unsigned Hybrid stays out of the headline even if its automated leg looks fine.
  • The gap between them is honest, not a defect: it is precisely the work that still needs a human decision. As an assessor, that gap is your worklist.

⚠What a finding looks like

A trust surface that only ever shows green teaches you nothing about its judgement. The whole point of an assertion is that it can say no. So every automated indicator here can be shown failing, on demand, without breaking anything real.

  • In Assessor, open any automated indicator and press Show a failing example. The same query and the same assertion run, but the evidence is swapped for a deliberately non-compliant example, so you watch a real finding form: a red FAILED verdict and a plain-language "why".
  • The failure is engine-computed, not hand-drawn. Each example is a broken evidence fixture run through the exact same engine/checks.py function that grades production; it is only shown if the real check genuinely returns false.
  • It is clearly labelled synthetic / teaching only and never touches the live posture count. Toggle Show the passing result to return to the real evidence.
  • The History tab shows the other half of the story: failures that were detected and then remediated back to validated over time. That pass → fail → pass arc is what an assessor should expect from a healthy, continuously-monitored system.

→How to run an assessment here

  • Start in Overview for posture: the bar and KPIs tell you the ratio of proven, pending, and human-dependent indicators at a glance.
  • Open Assessor and work an indicator top to bottom. Read the Input query first, does it actually measure what the pass criteria claims? Then read the Assertion, is the pass/fail logic faithful to the intent? Only then trust the Output.
  • Use Replay cycle when you want to walk a stakeholder through how a result was reached, not just that it passed.
  • Treat every Pending as a question, not a pass. Ask why it is not yet measured.
  • For Review and Manual items, check the attestation is dated, named, and within its review cadence before you accept it.

§Glossary

  • KSI — Key Security Indicator. A single testable statement of posture (id like KSI-SVC-SIN).
  • Collector — the read-only query that gathers evidence (Azure Resource Graph, Microsoft Graph, Sentinel/Log Analytics, or GitHub REST).
  • Assertion / pass-check — the Python function that turns evidence rows into pass or fail.
  • Evidence — the timestamped rows a collector returned, stored verbatim.
  • SDR — Security Decision Record. The signed run output this page is generated from.
  • Attestation — a dated human sign-off for Hybrid and Manual indicators.
